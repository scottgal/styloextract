The Complete Configuration Reference

This is the longest single page in the documentation set. It exists to exercise the structured walker under realistic load — many headings, prose paragraphs, fenced code blocks, lists, and tables — without falling off into the synthetic-corner-case territory that microbenchmarks reward.

Section one: lifecycle

The extraction lifecycle is the same on every request, regardless of which template path is taken. The parser produces a DOM, the cleaner removes scripts and tracking blobs, the fingerprinter computes a structural signature, and the segmenter / classifier produce the block list that the renderer turns into markdown.

The interesting question is how much of that work is amortised across requests with the same template signature. The answer is "most of it", which is why the fast-path threshold defaults to 0.85: tighter and you miss too many real matches; looser and you accept too many false ones.

  • Parse: AngleSharp DOM construction, dominated by tokeniser cost on large pages.
  • Clean: a single CSS selector pass removes a small fixed set of element kinds.
  • Fingerprint: MinHash over shingles plus an anchor-path sketch plus a pq-gram sample.
  • Segment: walk the body once, emit semantic tags and "blocky" divs as candidates.
  • Classify: score each candidate, prune overlapping subtrees, demote chrome.
  • Render: walk each block element subtree, emit structured markdown.

Most of the knobs you can turn live in the orchestration layer. The classifier itself is intentionally not configurable — its scoring constants are baked in and proven against the WCXB corpus.

Section three: code

The default service registration looks like this in C#:

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddStyloExtract(opts => {
    opts.FastPathThreshold = 0.85;
    opts.SlowPathThreshold = 0.75;
    opts.Refit.MissRatio = 0.35;
    opts.Refit.MinObservations = 5;
    opts.Refit.MinRules = 3;
});

var app = builder.Build();
app.UseStyloExtract();
app.Run();

And the equivalent in Python via the cross-language binding:

from styloextract import Extractor

ex = Extractor(
    fast_path_threshold=0.85,
    slow_path_threshold=0.75,
    refit_miss_ratio=0.35,
)

result = ex.extract(html, source_uri="https://example.com/page")
print(result.markdown)

Section four: more prose

Real documentation pages have far more prose than knobs. They exist to teach the reader how the system thinks, not just what each setting does. This section is where we add that prose so the benchmark page resembles a real reference page rather than a synthetic one.

Block extraction is fundamentally a structural problem. The classifier reads tag, class, id, link density, and text length. It does not read words. The proof is the WCXB F1 score: matching the rs-trafilatura baseline on English-language pages while running on any well-formed HTML document with no language-specific dependencies.

This generalises to multilingual pages, machine-generated SPA bodies, and the long tail of CMS templates that produce structurally-recognisable shapes even when their content is unique. The cost is that pages with no semantic structure — image-only carousels, table-only catalogues — are not great candidates for prose extraction. They are great candidates for shape-of-page fingerprinting, which is a different question entirely.

If you find yourself wishing the classifier read words, the right move is usually to add a deterministic post-filter on top of the block list rather than baking word-level logic into the classifier itself. The block list is your stable abstraction. Don't fight it.
