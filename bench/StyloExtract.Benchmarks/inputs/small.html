Understanding Dependency Injection in ASP.NET Core

By Alice Thompson June 15, 2026 8 min read

Dependency injection is a foundational pattern in modern .NET application development. When you register services in the container, the framework automatically resolves and injects them wherever they are needed. This decoupling makes components easier to test, maintain, and replace without touching the call sites that depend on them.

The built-in DI container in ASP.NET Core supports three service lifetimes: singleton, scoped, and transient. Choosing the correct lifetime for each service prevents subtle bugs that can be extremely difficult to diagnose in production. A common mistake is registering a stateful service as transient when it should be scoped to the current HTTP request.

Registering Services

Service registration happens in Program.cs via the IServiceCollection interface. You call extension methods like AddSingleton<T>, AddScoped<T>, or AddTransient<T> to bind an interface to its implementation. The container wires everything up before the application starts handling requests.

For larger applications you can group related registrations into extension methods on IServiceCollection. This keeps Program.cs readable and allows feature modules to register their own dependencies independently. The modular approach also makes it straightforward to swap implementations for testing purposes by registering a test double instead of the real service.

Constructor Injection

Constructor injection is the dominant pattern in ASP.NET Core applications. When the DI container constructs a controller or service, it inspects the constructor parameters and resolves each dependency from the container. If a dependency cannot be resolved, the container throws an exception at startup rather than at the point where the service would first be used.

Property injection is available but is generally discouraged because it makes dependencies optional in a way that hides required collaborators. Explicit constructor dependencies serve as self-documenting contracts: any class that needs a database context or an HTTP client must declare those requirements up front, making the dependency graph visible to anyone reading the code.

Testing with DI

The primary benefit of dependency injection for testing is the ability to substitute real services with mock or stub implementations. In xUnit or NUnit test projects you can build a minimal ServiceCollection, register the production services you want to test, replace the external dependencies with test doubles, and then resolve the subject under test from the container. The production code remains completely unaware of the substitution.
