Building Scalable Microservices with .NET and Kubernetes

Marcus Chen Senior Software Architect
June 10, 2026 22 min read
18.4k views 342 comments

Microservices architecture has transformed how teams build and deploy large-scale applications. By decomposing a monolithic application into independently deployable services, organizations gain the ability to scale individual components without touching the rest of the system, deploy changes at their own cadence, and isolate failures so that a bug in the payment service does not take down the entire catalog.

This tutorial walks through the full lifecycle of a microservices application built on .NET 8 and deployed to Kubernetes. We will cover service discovery, inter-service communication via gRPC and HTTP, distributed tracing with OpenTelemetry, and rolling deployments with zero-downtime strategies. By the end you will have a working reference architecture that you can adapt for real production workloads.

Before we dive into code, it is worth understanding why Kubernetes has become the de facto standard for container orchestration. Kubernetes provides a declarative API for describing the desired state of your cluster. You write YAML manifests that specify how many replicas of each service should run, what resources they require, and how they should be exposed to clients. The control plane continuously reconciles the actual state with the desired state, restarting failed pods and scaling deployments in response to load.

Project Structure and Service Boundaries

Defining service boundaries is the hardest part of microservices design and the most consequential. Boundaries drawn too coarsely produce distributed monoliths that retain all the operational complexity of microservices with none of the flexibility. Boundaries drawn too finely produce nano-services that spend most of their time on network round-trips and serialization rather than on actual business logic.

For our reference application we will build a simple e-commerce backend with four services: a Catalog service that manages product listings and inventory, an Orders service that handles the order lifecycle, a Payments service for processing transactions, and an Notifications service for sending transactional emails and push notifications. Each service owns its own database and communicates exclusively through well-defined APIs, never by sharing a database schema.

The project layout follows a standard multi-project solution structure. Each service lives in its own folder under src/Services/ and has its own .csproj. Shared domain types and interfaces go in a BuildingBlocks folder that all services reference. Integration tests for cross-service scenarios live under tests/Integration/ and use Testcontainers to spin up the required infrastructure.

Defining gRPC Contracts

We use gRPC for synchronous inter-service communication because Protocol Buffers give us strongly-typed contracts that are enforced at compile time. Any change to a .proto file that breaks a consumer surfaces as a build error rather than a runtime failure at three in the morning.

The Catalog service exposes a ProductService with methods for listing, getting, and updating product records. The .proto file lives in a shared Protos/ folder that is referenced by both the server-side implementation and all client assemblies. Code generation is wired into the build pipeline so that developers never need to run a separate codegen step.

For asynchronous communication we use a RabbitMQ message broker with MassTransit as the abstraction layer. When the Orders service successfully places an order, it publishes an OrderPlacedEvent that the Notifications service subscribes to. This publish-subscribe pattern decouples the two services: the Orders service does not know or care how many consumers exist, and adding a new consumer requires no changes to the publisher.

Service Discovery and Load Balancing

In Kubernetes, service discovery is built in. Each service exposes a stable DNS name of the form service-name.namespace.svc.cluster.local. The built-in kube-proxy implements basic round-robin load balancing across all healthy pods backing a service. For more sophisticated traffic management, including canary releases and circuit breaking, we use Istio as our service mesh.

Istio injects an Envoy sidecar proxy into every pod. All inter-service traffic flows through the sidecar, which enforces mutual TLS, collects telemetry, and applies traffic policies. The control plane lets you configure retries, timeouts, and circuit breakers via VirtualService and DestinationRule custom resources without changing any application code.

Client-side service discovery with a library like Steeltoe or a custom IServiceEndpointSelector is an alternative for environments where a service mesh is not available. The client queries a service registry on every request and caches the results with a short TTL. This approach adds complexity but gives you fine-grained control over load-balancing strategy and failure handling.

Distributed Tracing with OpenTelemetry

Debugging a distributed system without tracing is like debugging a monolith without a debugger. OpenTelemetry provides a vendor-neutral API for emitting traces, metrics, and logs from your application. The SDK ships as a NuGet package and integrates with ASP.NET Core via the AddOpenTelemetry() extension method.

Each incoming HTTP request or gRPC call automatically creates a root span. Downstream calls to other services or databases create child spans that inherit the parent trace context through HTTP headers. The complete trace is exported to a backend like Jaeger or Zipkin where you can visualize the full call graph, identify bottlenecks, and correlate spans across services by their shared trace ID.

We configure the tracer to sample 100% of traces in development and 5% in production. Tail-based sampling in the collector keeps costs manageable: we always retain traces that contained errors regardless of the sample rate. This ensures that every failed request is visible in the tracing backend even under high load.

Health Checks and Readiness Probes

Kubernetes uses liveness and readiness probes to decide when to restart a pod and when to include it in the load-balancing pool. ASP.NET Core exposes health endpoints via the AddHealthChecks() extension method. We register a liveness check that returns healthy as long as the process is running and a readiness check that verifies connectivity to the database and message broker before the pod starts receiving traffic.

The HealthCheckPublisher background service polls the health checks at a configurable interval and publishes the results to OpenTelemetry as gauge metrics. This lets you alert on degraded health before the pod fails its readiness probe and gets removed from the pool entirely, giving you an early warning that something is trending towards failure.

Deployments and Rolling Updates

We define each service as a Kubernetes Deployment resource with a RollingUpdate strategy. The maxUnavailable and maxSurge parameters control how many pods can be unavailable or in excess of the desired count during a rollout. Setting maxUnavailable: 0 and maxSurge: 1 ensures zero-downtime deploys at the cost of temporarily running one extra pod.

Container images are tagged with the Git commit SHA rather than a mutable tag like latest. This makes rollbacks trivial: to revert to the previous version you update the image tag in the deployment manifest and apply it. The rollout history is stored in Kubernetes so you can inspect the previous revision and compare resource definitions.

Resource requests and limits are specified for every container. Requests tell the scheduler how much CPU and memory to reserve for a pod when scheduling it onto a node. Limits cap the maximum resources the container can consume. Setting requests equal to limits gives predictable scheduling and prevents noisy-neighbour problems at the cost of potentially over-provisioning in exchange for guaranteed resource allocation.

Horizontal Pod Autoscaling

The Horizontal Pod Autoscaler adjusts the number of pod replicas based on observed metrics. The standard configuration scales on CPU utilization, but you can also scale on custom metrics exported via the Prometheus adapter. For the Orders service we scale on the length of the MassTransit consumer queue: when the queue depth exceeds a threshold, the autoscaler adds replicas to process messages faster.

KEDA, the Kubernetes Event-driven Autoscaling project, makes queue-based scaling straightforward. A ScaledObject custom resource references the RabbitMQ queue and specifies the target message count per replica. KEDA translates this into the standard Kubernetes autoscaling API, so all existing monitoring and alerting tooling continues to work without modification.

Configuration Management

Twelve-factor app methodology demands that configuration be stored in the environment rather than in code. In Kubernetes, configuration is stored in ConfigMap objects and injected into pods as environment variables or mounted as files. The IConfiguration API in ASP.NET Core reads from environment variables by default, so the transition from local development using appsettings.json to production using environment variables is transparent.

For configuration that changes frequently without a deployment, such as feature flags or rate limiting thresholds, a dedicated configuration service like Azure App Configuration, AWS AppConfig, or LaunchDarkly is preferable. The IOptionsMonitor pattern in ASP.NET Core makes it straightforward to react to configuration changes at runtime without restarting the application. The options class listens for change notifications and updates the in-memory representation, which is then picked up by the next request to use that configuration value.

Avoid the temptation to store large amounts of configuration in environment variables. A pod with dozens of environment variables becomes difficult to reason about and test. For large configuration payloads, use a mounted ConfigMap volume. The mounted volume approach also supports configuration hot-reloading: when the ConfigMap is updated, Kubernetes updates the mounted file after a short propagation delay, which your application can detect and apply without a restart.

Pod Disruption Budgets

A Pod Disruption Budget specifies the minimum number of replicas that must remain available during voluntary disruptions such as node drains, cluster upgrades, or manual pod evictions. Without a disruption budget, a node drain could take down multiple replicas simultaneously, causing a service outage. A budget with minAvailable: 1 ensures that at least one replica is running throughout any disruption.

Disruption budgets interact with the autoscaler: if the autoscaler scales down to fewer replicas than the disruption budget allows, the scale-down is blocked. This is intentional behavior; the budget represents an operator guarantee about service availability. If you find the budget is blocking legitimate scale-downs, review whether the budget thresholds are still appropriate for your current traffic levels.

Resource Quotas and Limit Ranges

Resource quotas constrain the total resources that can be consumed by all pods in a namespace. They prevent a single team from monopolizing cluster capacity and help enforce capacity planning discipline. A namespace with a CPU quota of four cores and a memory quota of 8GB cannot schedule more pods than those limits allow, regardless of what the underlying nodes can provide.

Limit ranges set default resource requests and limits for pods that do not specify them explicitly. Without a limit range, pods without resource limits can consume unbounded amounts of CPU and memory, which causes noisy-neighbour problems and makes capacity planning impossible. Setting sensible defaults catches the common case of developers forgetting to add resource limits to their manifests.

Secrets Management

Never bake secrets into container images or environment variable literals in your Kubernetes manifests. Instead, store secrets in a dedicated secrets manager such as HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault. The External Secrets Operator synchronizes secrets from the external store into Kubernetes Secret objects that your pods can consume as environment variables or mounted volumes.

For lower-complexity deployments, Kubernetes Secrets combined with RBAC is often sufficient. The key is ensuring secrets are rotated automatically and that the rotation is reflected in running pods without manual intervention. Vault's dynamic secrets feature generates short-lived credentials on demand, eliminating the problem of long-lived credentials leaking through logs or error messages.

Workload identity, available in managed Kubernetes offerings like GKE Workload Identity and AWS EKS IAM Roles for Service Accounts, is the cleanest solution for cloud services: each pod assumes an IAM role and can access cloud resources directly without any static credentials in the cluster at all.

Networking and Ingress

Kubernetes services are only reachable from within the cluster by default. To expose services to external clients you define an Ingress resource backed by an ingress controller such as nginx, Traefik, or the cloud provider's native load balancer integration. The ingress controller terminates TLS, routes traffic by hostname and path, and optionally handles rate limiting and authentication.

For HTTP-only traffic, an Ingress is sufficient. For more complex routing scenarios, such as routing to different services based on gRPC service name or supporting WebSocket long-lived connections, you may need a Gateway API resource instead. The Kubernetes Gateway API is the successor to the legacy Ingress API and provides a richer, more extensible model for defining traffic routing rules.

Network policies restrict which pods can communicate with each other. By default, all pods in a namespace can reach all other pods in the cluster. A default-deny network policy combined with explicit allow rules for known communication paths gives you defense-in-depth: even if an attacker compromises one pod, they cannot trivially reach the database or the secrets manager.

Observability Stack

A complete observability stack for Kubernetes includes metrics, logs, and traces. For metrics we use Prometheus for collection and Grafana for visualization. The kube-state-metrics and node-exporter projects expose cluster-level metrics that supplement the application metrics your services emit. Grafana dashboards are checked into version control alongside the application code so that developers can iterate on them without touching production.

Log aggregation with Loki and Grafana provides a unified query interface for logs across all services. Structured logging with JSON output is essential: unstructured log lines are difficult to query efficiently at scale. We configure the Microsoft.Extensions.Logging provider to write structured JSON to stdout, and Loki's Promtail agent scrapes the pod logs and indexes them for querying.

Traces from OpenTelemetry are exported to Tempo, which integrates natively with Grafana. The combination of Prometheus metrics, Loki logs, and Tempo traces in a single Grafana workspace enables correlated investigation: when an alert fires on a spike in error rate, you can click through from the metric to the relevant log lines and from a log line to the full distributed trace, all within the same UI.

CI/CD Pipeline

The CI/CD pipeline builds Docker images, runs unit and integration tests, pushes images to the container registry, and applies Kubernetes manifests. We use GitHub Actions for CI and Argo CD for continuous delivery. The key principle is that the pipeline never applies manifests directly to the cluster; instead, it commits updated image tags to a GitOps repository that Argo CD watches.

Argo CD continuously reconciles the cluster state with the declarative manifests in the GitOps repository. When a new image tag is committed, Argo CD detects the drift and applies the updated manifest. The audit trail lives in the Git history: every deployment is a commit, and rolling back is as simple as reverting the commit and letting Argo CD re-sync.

Pre-deployment validation with kubeval and conftest catches manifest errors before they reach the cluster. Policy enforcement with OPA Gatekeeper enforces organizational standards such as requiring resource limits on all containers and prohibiting :latest image tags in production. These gates run as admission webhooks and reject non-compliant resources at the API server.

Database Patterns for Microservices

Each microservice owns its own persistent store. Sharing a database between services is an anti-pattern: it creates a hidden coupling at the schema level and makes it impossible to evolve the services independently. When two services write to the same table, a schema change for one service must be coordinated with the other, negating the independence that microservices are supposed to provide.

Choosing the right database technology for each service is a significant advantage of the microservices model. A catalog service that serves high read volumes with infrequent writes benefits from a read-optimized store like PostgreSQL with materialized views and aggressive caching. A session store benefits from a key-value store like Redis with TTL-based expiration. A time-series metrics store benefits from InfluxDB or TimescaleDB. The database-per-service pattern lets you pick the right tool for each job.

The outbox pattern ensures reliable message delivery when a service needs to update its database and publish an event atomically. Without the outbox, a service might update the database and then fail before publishing the event, leaving the system in an inconsistent state. The outbox pattern writes the event to a table in the same database transaction as the business data change. A separate relay process reads from the outbox table and publishes messages to the broker, deleting each record after successful publication. Because the relay uses at-least-once delivery, consumers must be idempotent.

Error Handling and Resiliency Patterns

In a distributed system, partial failures are inevitable. A service may be temporarily unavailable, a network partition may separate pods, or a downstream dependency may respond with intermittent errors. Your services must be resilient to these scenarios without propagating failures to their callers. The three canonical resiliency patterns are retry with backoff, circuit breaking, and bulkhead isolation.

The Polly library provides all of these patterns for .NET applications. A retry policy automatically retries failed requests with exponential backoff and jitter. The jitter term is important: without it, all retrying clients synchronize their retry attempts, causing a thundering-herd effect that can overwhelm a recovering downstream service. Add jitter to spread retries over a time window.

A circuit breaker stops sending requests to a failing service after a threshold of consecutive failures is exceeded. The circuit transitions from closed (passing requests) to open (rejecting requests with a fast-fail response) when the threshold is hit. After a configurable timeout the circuit transitions to half-open, allowing a probe request through. If the probe succeeds, the circuit closes; if it fails, the circuit stays open and the timeout resets.

Bulkhead isolation limits the number of concurrent requests to a downstream dependency. Without bulkheads, a slow dependency can exhaust the thread pool or connection pool of the calling service, causing it to appear down even though the dependency is merely slow rather than unavailable. Separate thread pools or semaphore-based concurrency limits for different downstream services prevent one slow dependency from bringing down the entire service.

Testing Microservices

Unit tests for individual service components remain straightforward: mock the dependencies, exercise the behavior, assert the outcomes. The complexity arises when you need to test behaviors that span service boundaries. Integration tests that exercise a real service instance against real infrastructure dependencies are essential for catching issues that unit tests cannot reveal.

Testcontainers eliminates the need for a running Kubernetes cluster in CI. Each integration test spins up the required infrastructure containers (PostgreSQL, RabbitMQ, Redis) via the Docker API, runs the tests, and tears the containers down. The containers are isolated per test run, so parallel test execution does not cause interference. Startup time is typically a few seconds per container, making this approach practical even in fast CI pipelines.

Contract testing with Pact ensures that a consumer's expectations of a provider's API are met without requiring both services to be running simultaneously. The consumer generates a pact file describing the interactions it expects, and the provider verifies that its actual implementation satisfies the pact. This catches API contract violations early without end-to-end integration tests that require the entire system to be running.

End-to-end tests against a fully deployed environment provide the final confidence layer. These tests run a realistic user journey from start to finish and verify observable outcomes. They are necessarily slower and more brittle than unit or integration tests, so their number should be limited to the most critical user journeys. The goal is not comprehensive coverage but a smoke test that validates the integration wiring.

Load testing should be part of every service's pre-production validation gate. A service that passes all functional tests may still fail under realistic traffic patterns because of database connection pool exhaustion, garbage collection pauses, or lock contention in shared resources. K6 and NBomber are excellent load testing tools with first-class support for .NET and a scripting model that lets you define realistic user scenarios with think time and data variation.

Chaos engineering, popularized by Netflix Chaos Monkey, deliberately injects failures into production systems to validate their resilience claims. Start with a defined steady state, introduce a controlled perturbation such as killing a random pod or injecting network latency, and observe whether the system returns to steady state within an acceptable time window. Chaos Mesh for Kubernetes provides a Kubernetes-native interface for injecting various failure types including pod failures, network partitions, and I/O errors.

Multi-Tenancy and Namespace Strategy

Namespaces provide a basic level of isolation between different teams or environments within a single cluster. A common pattern is to use one namespace per environment per service, with RBAC rules that prevent developers from applying manifests to production namespaces directly. All production changes go through the CI/CD pipeline, which enforces code review and automated testing before deployment.

For stronger isolation, such as when multiple customers share a cluster, you can use virtual clusters with tools like vCluster. Each tenant gets a virtual Kubernetes API server that appears to the tenant as a fully independent cluster but shares the underlying node resources. This provides tenant isolation at the API level without the operational overhead of provisioning a separate physical cluster for every tenant.

Conclusion

Microservices on Kubernetes is not a silver bullet. It introduces genuine operational complexity around service discovery, distributed tracing, and deployment coordination that does not exist in a well-structured monolith. Teams should adopt microservices only when they have genuinely independent deployment cadences or scaling requirements that make the added complexity worthwhile.

That said, the combination of .NET, Kubernetes, and OpenTelemetry provides an excellent foundation for building and operating large-scale distributed systems. The ecosystem is mature, the tooling is excellent, and the community has produced extensive guidance on patterns for resilience, observability, and security. The reference architecture in this tutorial is a starting point you can extend as your requirements evolve.
